# 报数合计 ↔ 中通明细单一数据链设计

## 目标

把当前系统中与当期中通发货有关的核心链路收敛为一条清楚、可校验、可追溯的数据链，确保：

1. 报数编辑页中“中通物流公司”合计有唯一对应的ZTO-MF总份数。
2. 报数确认时的校验逻辑稳定、明确，可记录一致或不一致的结果。
3. ZTO-MF的后续修改、以及每次导出，都能够追溯到确认时和导出时使用的数量状态。

本次设计**不**把“收件人/订阅”作为主链的自动生成来源接入。物流管理和订阅管理暂时保留现状，等待后续订单管理系统上线后再作为上游接入。

## 当前问题

当前网页中存在两套与发货相关的入口和数据路径：

1. **印数管理 → 发货**  
   这是旧的发货预览页，当前依赖 `shipping_records` 这条链。

2. **物流管理 → ZTO-MF**  
   这是当前维护中通明细的执行页面，依赖 `shipping_details`。

同时，报数确认时实际拿来比对的是 `shipping_details` 的合计，而旧发货页和导出逻辑又仍然使用 `shipping_records`。结果是：

- 用户在网页上会看到两套“发货明细”入口。
- “报数确认校验”和“发货导出来源”不是同一条链。
- 确认后如果中通明细发生变化，很难回答“确认时到底按哪个数量通过的”。

## 本次范围

### 包含

1. 建立“报数编辑页 ↔ ZTO-MF页”的单一链路。
2. 统一报数确认时的比对来源。
3. 强化中通明细修改留痕和确认留痕。
4. 为报数导出和中通导出增加可追溯快照。
5. 处理旧“发货”页面的去主流程化。

### 不包含

1. 不做“收件人/订阅 → 中通明细自动生成”。
2. 不做订单管理系统。
3. 不做收件人、订阅和中通明细之间的实时双向同步。

## 用户视角下的页面职责

### 1. 物流管理 → 收件人

该页继续作为收件人和订阅的管理入口保留，但不参与本次主链改造。

用户可继续：

- 新增/编辑收件人
- 停发/恢复收件人
- 在“订阅”抽屉中维护订阅记录

本次只需要在页面文案上明确：这里管理的是基础资料和订阅规则，不是当前期的中通执行明细。

### 2. 物流管理 → ZTO-MF

该页定义为**当前阶段唯一的中通执行面**。

用户在这里完成：

- 选择期号
- 查看当期ZTO-MF
- 新增、编辑、删除单条明细
- 批量修改状态或截止日期
- 查看操作日志
- 查看当期总份数
- 导出当期中通明细

页面顶部显示的“合计 X 份”，定义为**当期中通执行数量**。报数确认时只认这个值。

### 3. 印数管理 → 报数编辑页

该页继续作为报数输入和确认入口。

用户在这里完成：

- 编辑报数项
- 查看“发货目的地汇总”
- 导出报数文件
- 点击“确认报数”

点击“确认报数”时，系统只比较：

- 报数页中去向为“中通物流公司”的合计
- 当前期号的ZTO-MF总份数

不再从旧的 `shipping_records` 链路取值。

### 4. 印数管理 → 发货

该页当前属于旧链入口。

本次建议将它从主流程中移除，采用以下方式之一：

1. **推荐：改为跳转页**  
   点击后直接跳转到“物流管理 → ZTO-MF”，并自动定位到当前期号。

2. **备选：改为只读页**  
   保留页面，但只显示只读说明，并提示用户到“ZTO-MF”页维护。

推荐方案是跳转页，因为它能尽快消除“双入口”造成的理解负担。

## 单一数据链定义

本次收敛后的主链为：

`印数管理 → 报数编辑页（中通合计） ↔ 物流管理 → ZTO-MF（当期总份数）`

其中：

- **报数编辑页** 是报数侧输入和确认入口。
- **ZTO-MF页** 是当前期中通执行明细的唯一事实来源。
- **确认记录 / 导出快照** 是追溯层。

收件人、订阅暂时不在这条主链中承担自动生成职责。

## 校验设计

### 校验时机

在“报数编辑页”点击“确认报数”时执行。

### 校验规则

固定比较两个值：

1. 报数中去向为“中通物流公司”的合计份数
2. ZTO-MF页当前期号的合计份数

### 校验结果

#### 一致

- 允许确认
- 记录“本次确认校验通过”

#### 不一致

延续当前业务规则：**允许确认，但必须强提示并留痕**。

提示内容必须明确包含：

- 报数中通合计
- ZTO-MF合计
- 差值

并记录这是一次“不一致确认”。

### 页面交互

在确认弹窗或确认结果提示中，直接向用户展示上述三个数字，而不是只提示“请核查”。

## 留痕与追溯设计

### 1. 中通明细操作留痕

“ZTO-MF”页已有新增、编辑、删除、批量修改的操作日志。

本次保留现有日志机制，并补充一个更高层的追溯目标：

- 系统需要知道“报数确认时对照的是哪一版中通明细”
- 如果确认后中通明细又被改动，系统能区分：
  - 确认时的数量
  - 当前最新数量

### 2. 报数确认留痕

每次确认时都应记录至少这些信息：

- 期号
- 确认时间
- 报数中通合计
- 中通明细合计
- 是否一致
- 差值

如果是不一致确认，也要被明确标记。

### 3. 导出快照

每次导出都生成一条可追溯记录，而不是只依赖“当前库里是什么值”。

记录内容至少包括：

- 导出时间
- 导出类型（报数 / 中通）
- 期号
- 导出时的报数中通合计
- 导出时的中通明细合计
- 是否一致

这样后续可以回答：

- 某次发给外部的文件，是按哪个数量导出的
- 导出时是否已经一致
- 导出后是否还有人改过中通明细

## 改造后的用户流程

### 日常操作流程

1. 用户进入“物流管理 → ZTO-MF”，维护当期中通明细。
2. 用户进入"印数管理 → 报数编辑页"，完成报数输入。
3. 用户点击“确认报数”。
4. 系统比较报数中通合计和当期中通明细总份数。
5. 系统记录本次确认结果。
6. 用户分别在报数页和中通明细页执行导出。
7. 系统为每次导出记录快照。

### 异常操作流程

1. 用户确认报数后，又去修改了ZTO-MF。
2. 系统保留明细修改日志。
3. 系统能追溯到：
   - 报数确认时使用的中通数量
   - 修改后的当前中通数量
4. 后续再次导出时，新的导出快照使用修改后的数量状态。

## 实施原则

1. **单一执行面**  
   当前期ZTO-MF只认“物流管理 → ZTO-MF”页面。

2. **单一校验来源**  
   报数确认时只与当期ZTO-MF合计比对。

3. **校验可解释**  
   一致或不一致都要把数值讲清楚。

4. **结果可回看**  
   确认和导出都要保留可回看的记录。

5. **先解决主链，再接上游**  
   先把“报数 ↔ 中通明细”主链理顺；收件人/订阅未来再通过订单系统接入。

## 风险与约束

1. 当前系统里旧“发货”页仍然存在，如果不处理，会继续造成用户混淆。
2. 如果只保留明细操作日志、不记录确认时的对照结果，仍然无法完整回答“当时按哪个数量确认”的问题。
3. 如果导出不留快照，只看当前数据，也无法准确还原历史对外文件对应的数量状态。

## 推荐结论

本次不追求把收件人、订阅、订单一口气全部打通，而是聚焦完成以下目标：

**让“报数编辑页确认时的中通合计”和“ZTO-MF页当前期号总份数”形成唯一、稳定、可追溯的对应关系。**

这是当前阶段最小、最稳、最能直接解决业务问题的方案，也是后续接入订单管理系统的正确前置条件。
