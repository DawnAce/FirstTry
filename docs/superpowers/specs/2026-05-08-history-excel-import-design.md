# 历史期数 Excel 补录设计

## 问题

当前首页支持为过期期数补录报数，但创建后仍依赖人工逐项录入。用户希望提供一个 Excel 文件，让系统按现有 `2649` 期的报数编辑页结构预填历史数据，再由人工检查并通过现有“确认报数”流程提交。

## 目标

为“其他期数补录”增加一条 **创建草稿 -> 导入 Excel -> 进入可编辑页 -> 手动确认提交** 的工作流。

导入后需要尽量恢复以下信息：

1. 报数页面中的各项数值
2. 临时加印归属明细
3. 版数
4. 备注

缺失字段不阻止补录，保留为空并允许继续人工编辑。

## 方案选择

最终采用方案 1：

- 在首页创建过期期数草稿
- 跳转到该期报数编辑页
- 通过 Excel 导入预填数据
- 保持草稿状态
- 由用户手动确认报数

未采用的方案：

- 直接从 Excel 自动确认：风险高，缺少人工复核
- 单独做历史补录向导：成本更高，与现有编辑流重复

## 交互设计

### 首页

- 继续保留“其他期数补录”的期数选择与“创建”按钮
- 创建过期期数时，行为与当前一致：先生成该期草稿
- 创建成功后直接跳转到该期报数编辑页

### 报数编辑页

- 在页面顶部新增 Excel 导入入口
- 用户选择 Excel 后，系统解析并写入当前草稿
- 导入完成后页面立即刷新为预填状态
- 未识别字段在页面中以“未导入项”形式提示，但不阻止后续编辑
- 页面仍保持现有草稿编辑体验与现有“确认报数”按钮

## 导入规则

### 数据来源

仅支持 Excel 文件，不支持 PDF 或图片识别。

### 映射策略

- 以内置的“2649 期页面结构映射”为基准
- 优先按表头、项目名、区块标题进行语义匹配
- 不把固定行号作为唯一依据，避免模板轻微变动导致整体失败

### 字段处理

- **报数项数值**：匹配后写入对应 `report_entries`
- **临时加印归属明细**：解析为多行明细，写入 `temp_print_details`
- **版数**：解析成功则更新 `issues.page_count`
- **备注**：解析成功则更新 `issues.notes`
- **缺失字段**：保留为空，不自动填默认值
- **无法识别字段**：记录为未导入项并反馈到前端

### 状态约束

- 导入仅对草稿期数生效
- 导入成功后不自动确认
- 仍由用户手动触发现有确认流程

## 后端设计

### 新能力

新增一个“历史 Excel 导入”接口，负责：

1. 接收当前期数与上传文件
2. 解析 Excel
3. 按映射结果更新当前期的报数数据
4. 同步临时加印归属明细
5. 返回导入摘要与未导入项

### 服务拆分

建议拆成两个清晰职责：

- **Excel 解析服务**：只负责读取文件并产出标准化结果
- **导入应用服务**：负责把结果写入 `issues`、`report_entries`、`temp_print_details`

这样可以让解析规则与数据库写入分离，便于后续维护模板兼容性。

## 前端设计

### 组件行为

- 在 `ReportEditor` 顶部增加导入控件
- 上传成功后刷新当前期详情与临时加印明细
- 展示导入成功摘要，例如：
  - 导入报数项数量
  - 导入临时加印明细数量
  - 是否更新版数/备注
  - 未导入项列表

### 缓存策略

- 导入完成后失效当前期报数相关查询
- 同步刷新临时加印归属明细查询
- 确保页面显示与数据库一致

## 错误处理

- 文件格式错误：提示仅支持 Excel
- 文件内容不可识别：提示导入失败并说明原因
- 部分字段识别失败：按“部分成功”处理，保留已导入内容并提示未导入项
- 非草稿状态导入：明确提示不可导入

## 测试与验证

### 核心验证点

1. 创建过期期数后能进入可编辑报数页
2. 导入 Excel 后，报数项正确落到对应输入项
3. 临时加印归属明细能按多行恢复
4. 版数和备注能正确写入
5. 缺失字段保持为空，不阻塞编辑
6. 未导入项会明确展示
7. 导入后仍需用户手动确认

### 非目标

- 不做 OCR
- 不支持 PDF/图片导入
- 不在本次范围内支持多套 Excel 模板配置 UI

## 实施边界

本次实现聚焦“历史补录 Excel 导入”最小闭环，不改动现有确认、作废、导出等主流程，只在草稿补录链路中新增导入能力。
