# 往期模板导入设计

## 背景

当前系统已经支持：

1. 为未创建期号生成草稿期数
2. 进入报数编辑页继续修改
3. 维护中通发货明细
4. 走现有确认、发货、导出流程

但系统还不支持“给一组往期资料，直接生成该期完整页面”的能力。

此前的 `2026-05-08-history-excel-import-design.md` 侧重“为单个草稿期数导入一份原始报数 Excel”。本次需求已经收敛为另一条更稳定的路线：**优先支持系统模板导入，而不是直接解析原始历史 Excel**。

## 目标

为系统新增一条完整的“往期导入”工作流，使用户可以通过两份系统模板文件一次性生成某一期的完整草稿数据：

1. 报数模板
2. 中通发货明细模板

导入成功后，系统应：

1. 创建目标期号的 `Issue`
2. 写入该期的 `ReportEntry`
3. 写入该期的 `TempPrintDetail`
4. 写入该期号对应的 `ShippingDetail`
5. 保持草稿状态
6. 跳转到现有报数编辑页，由用户继续复核和修改

## 非目标

本次不做以下内容：

1. 不直接解析用户现有的复杂原始 Excel
2. 不做 OCR、PDF、图片导入
3. 不在导入页内提供整表编辑器
4. 不在本次范围内支持“重复导入后覆盖已有期数”
5. 不自动确认报数

## 方案结论

最终采用以下方案：

1. 在"印数管理"增加 **「导入往期」** 入口
2. 点击后进入独立的“往期导入”页面，而不是弹窗
3. 一次导入任务中分别上传两类模板文件
4. 后端先执行预检，再执行正式导入
5. 两份文件必须属于同一期，且该期在系统中尚未存在
6. 导入页只负责上传、识别、校验、确认、生成
7. 细节修改继续回到现有报数页和相关页面完成

## 用户工作流

### 步骤 1：进入导入页

用户从"印数管理"点击 **「导入往期」**，进入独立页面。

### 步骤 2：上传两份模板文件

用户在同一次导入任务中分别上传：

1. 报数模板文件
2. 中通发货明细模板文件

两份文件属于同一次业务导入，但在页面上分开选择和展示状态。

### 步骤 3：系统预检

系统上传后立即执行预检，输出：

1. 从报数模板识别出的期号和出版日期
2. 从中通模板识别出的期号
3. 报数项数量
4. 临时加印明细数量
5. 发货明细数量
6. 是否同一期
7. 该期是否已存在
8. 是否允许继续导入

### 步骤 4：用户确认

导入页展示识别摘要和校验结果，用户确认无误后点击 **「导入并生成」**。

第一版只允许用户确认结果，不在该页逐项编辑明细。

### 步骤 5：正式导入

后端在一个事务中完成正式写入。成功后：

1. 生成目标期数草稿
2. 返回该 `issue_id`
3. 前端跳转到现有报数编辑页

## 模板设计

第一版采用“系统模板优先”策略。模板由系统提供，字段和现有数据结构保持一致，但表头采用更适合人工整理的中文名称。

### 报数模板

建议为一个 Excel 工作簿，包含以下工作表：

#### 1. 基本信息

字段：

1. 期号
2. 出版日期
3. 版数
4. 备注

其中：

- 期号、出版日期为必填
- 版数默认允许为空，导入时为空则回退到系统默认值 `24`
- 备注可为空

#### 2. 报数项

字段：

1. 分类编码
2. 分类名称
3. 项目名称
4. 数值

说明：

- 第一版建议由系统直接生成一份固定结构模板，预置当前系统中的全部报数项
- 用户只需要填写或粘贴数值，不需要自行设计行结构
- 导入时以 `category + sub_category` 作为落库定位键
- 分类名称只用于人工可读性，不作为唯一定位依据

#### 3. 临时加印明细

字段：

1. 部门
2. 自定义名称
3. 加印数量
4. 自留数量

说明：

- 该工作表允许为空
- 导入时写入 `temp_print_details`
- 同时同步 `social_use / 临时加印_自留` 对应的报数值

### 中通发货明细模板

建议为一个 Excel 工作簿，包含以下工作表：

#### 1. 基本信息

字段：

1. 期号
2. 出版日期（可选）

说明：

- 期号必填
- 出版日期用于辅助人工核对，可选

#### 2. 发货明细

字段：

1. 工作表名称
2. 渠道
3. 子渠道
4. 运输方式
5. 频次
6. 状态
7. 姓名
8. 地址
9. 电话
10. 数量
11. 截止日期
12. 备注
13. 附加信息
14. 城市
15. 网点名称
16. 网点大厅
17. 联系人
18. 序号
19. 期数
20. 公司

说明：

- 该模板字段与 `shipping_details` 现有结构对齐
- 此处“期数”指 `period_count` 业务字段，不用于识别导入目标期号；导入目标期号统一来自模板“基本信息”
- `confirmation`、`shipped_at` 不作为模板导入字段，由系统后续业务流程维护
- 可为空字段允许留空
- `transport`、`frequency`、`status` 可以由模板显式填写；若空值则使用系统默认值

## 前端设计

### 路由与页面

新增独立页面，例如：

- `/history-import`

页面分为四个顺序区块：

1. 上传文件
2. 自动识别
3. 校验与确认
4. 导入结果

采用长页面而不是弹窗，原因如下：

1. 该流程步骤较多，信息密度高
2. 需要展示上传状态、识别摘要、错误信息、导入进度
3. 后续若增加模板下载、导入历史、重新上传等能力，独立页面更易扩展

### 页面行为

导入页需要具备以下能力：

1. 分别上传两份模板
2. 展示每份文件的文件名、状态、替换操作
3. 触发预检并展示结果
4. 在预检成功前禁用“导入并生成”
5. 正式导入中展示进度和禁用重复提交
6. 导入成功后跳转到对应报数编辑页

### 模板下载

导入页应提供两份模板下载入口：

1. 下载报数模板
2. 下载中通发货明细模板

其中报数模板建议由后端根据当前系统报数结构动态生成，以保证与现有 `report_entries` 结构一致。

## 后端设计

### 接口设计

建议新增一组独立的导入接口，而不是挂在现有 `issues` 或 `reports` 资源下。

#### 1. 模板下载接口

- `GET /api/history-import/templates/report`
- `GET /api/history-import/templates/shipping`

职责：

1. 返回标准模板文件
2. 报数模板包含当前系统预置报数项结构
3. 中通模板包含固定表头和示例说明

#### 2. 预检接口

- `POST /api/history-import/preview`

请求内容：

1. 报数模板文件
2. 中通发货明细模板文件

响应内容：

1. 识别出的期号和出版日期
2. 报数项数量
3. 临时加印明细数量
4. 发货明细数量
5. 校验结果
6. 可读的错误信息
7. 一个短期有效的 `import_session_id` 或 `import_token`

预检阶段不写正式业务数据。

#### 3. 正式导入接口

- `POST /api/history-import/commit`

请求内容：

1. `import_session_id` 或 `import_token`

响应内容：

1. 新建期数的 `issue_id`
2. `issue_number`
3. 导入摘要

### 服务拆分

建议拆分为三层：

1. **模板生成服务**
   - 负责输出标准 Excel 模板
2. **模板解析服务**
   - 负责读取上传文件并转成标准化结构
3. **历史导入应用服务**
   - 负责预检校验、事务写入和最终结果返回

### 事务规则

正式导入必须在单个事务中完成以下写入：

1. 创建 `Issue`
2. 写入 `ReportEntry`
3. 写入 `TempPrintDetail`
4. 写入 `ShippingDetail`

任一步失败都必须整次回滚，不得留下半成品数据。

### 预检缓存

预检成功后，后端保存一次短期有效的标准化结果。正式导入只消费这次预检结果，不重复解析文件。

这样可以避免：

1. 用户确认前后解析结果不一致
2. 重复解析带来的性能浪费
3. 正式导入阶段重新上传文件的复杂交互

## 数据落库规则

### Issue

创建新的 `Issue` 记录：

1. `issue_number` 使用报数模板识别结果
2. `publish_date` 使用报数模板识别结果
3. `page_count` 使用报数模板中的版数，缺失时默认 `24`
4. `notes` 来自报数模板，允许为空
5. `status` 固定为 `draft`

### ReportEntry

导入时不沿用“复制上一期”逻辑，而是直接按模板结果写入该期全部报数项。

前提是报数模板由系统生成，因此其报数项结构与当前系统定义保持一致。

### TempPrintDetail

若模板中存在临时加印明细，则逐行写入。

写入后需要同步 `social_use / 临时加印_自留` 的值，保持与现有报数逻辑一致。

### ShippingDetail

逐行写入目标 `issue_number` 下的中通发货明细。

第一版不在导入阶段生成 `ShippingRecord`。这部分继续沿用现有进入发货页时的生成逻辑。

## 校验规则

### 必须通过的校验

1. 两份文件都已上传
2. 两份文件格式正确
3. 报数模板可识别出期号和出版日期
4. 中通模板可识别出期号
5. 两份文件识别出的期号一致
6. 目标期号在系统中不存在
7. 报数项结构与系统定义一致
8. 关键数值字段可解析为整数

### 默认处理规则

1. **禁止重复导入**
   - 若该期已存在，直接阻止导入
2. **禁止跨期混导**
   - 若两份文件不是同一期，直接阻止导入
3. **禁止部分成功**
   - 任一关键校验失败，正式数据一律不写入

## 错误处理

前端只展示明确失败原因，不做模糊成功提示。

第一版应覆盖的典型错误包括：

1. 该期已存在，不能重复导入
2. 两份文件不是同一期
3. 报数模板无法识别
4. 中通模板无法识别
5. 报数模板缺少关键字段
6. 中通模板缺少关键字段
7. 预检结果已失效，请重新上传

## 测试与验收标准

### 核心验收标准

至少满足以下结果：

1. 用户可以从"印数管理"进入"导入往期"页面
2. 用户可以下载两份系统模板
3. 使用合法模板导入后，系统成功生成目标期数并跳转到报数编辑页
4. 报数页条目和值与模板内容一致
5. 临时加印明细导入后可在现有报数相关逻辑中正常使用
6. 中通发货明细可按该期号查到，数量与模板一致
7. 任一校验失败时，不写入任何正式业务数据
8. 导入完成后仍可继续使用现有确认、发货、导出流程

### 推荐验证样本

为了后续实现和验收，建议准备：

1. 至少一组确认无误的模板样本，作为标准验收数据
2. 至少一组包含临时加印明细的模板样本
3. 至少一组中通明细含空字段的模板样本

## 后续扩展方向

本次方案故意选择“模板导入优先”的窄口，确保第一版稳定可用。后续若有需要，可以在此基础上继续扩展：

1. 增加“原始 Excel -> 系统模板”的转换工具
2. 增加导入历史记录与失败明细下载
3. 增加更多模板版本兼容
4. 增加导入前差异预览
