# 年度刊期表上传与保存设计

## 背景

系统当前通过 `publication_schedule` 保存结构化刊期数据，并用 2026 年种子文件初始化刊期表。管理员每年会拿到正式 PDF 刊期表，希望直接上传、核对并保存，后续创建报数期次时自动使用对应年份的刊期数据。

本设计以 2026 年《中国经营报》刊期表 PDF 为基准：全年 52 个周一计划，实际出版正报 49 期，休刊 3 期，期号范围 2635-2683，休刊周不占用期号。

## 目标

- 支持上传年度刊期表 PDF，并解析为结构化刊期数据。
- 保存原始 PDF 上传记录，方便追溯年度刊期来源。
- 保存前提供只读预览和人工核对，避免 PDF 识别错误直接进入生产数据。
- 保存后驱动现有 Dashboard、创建期次、当年期次序号计算等流程。
- 在该年已有报数期次时，阻止会破坏既有期次对应关系的覆盖操作。

## 非目标

- 不自动创建 `issues` 报数期次；上传刊期表只维护年度计划。
- 不替代现有报数表、发货明细、往期导入流程。
- 首版不要求支持任意报刊版式，只针对当前《中国经营报》刊期表形态做稳定识别。
- 首版不引入第三方 OCR；依赖 PDF 可抽取文本。扫描件可在后续版本支持。

## 方案选择

采用“PDF 自动识别 + 预览确认 + 保存原文件和结构化数据”。

备选方案中，“只存 PDF”无法驱动创建期次；“只支持 Excel 模板”与当前业务收到 PDF 的实际流程不匹配；“PDF + Excel 兜底”更完整但首版开发量更大，可在 PDF 解析稳定后追加。

## 后端设计

### 数据模型

继续使用 `publication_schedule` 作为最终刊期计划表：

- `year`
- `issue_number`
- `publish_date`
- `is_suspended`

新增上传记录表，例如 `publication_schedule_uploads`：

- `id`
- `year`
- `original_filename`
- `stored_path`
- `status`：`previewed`、`committed`、`failed`
- `summary_json`：解析摘要，如出版期数、休刊数、期号范围、备注
- `error_json`：解析或校验错误
- `uploaded_by`
- `created_at`
- `committed_at`

原始 PDF 存放在后端受控目录中，例如 `backend/uploads/publication_schedules/{year}/`，数据库只保存相对路径。

### API

- `GET /api/schedule?year=YYYY`
  - 延续现有查询能力，按年份返回最终刊期数据。
- `GET /api/schedule/uploads`
  - 返回年度上传记录列表，可按年份过滤。
- `POST /api/schedule/uploads/preview`
  - 接收 PDF 文件。
  - 抽取 PDF 文本并解析为刊期行。
  - 保存原始 PDF 和上传记录，但不写入最终 `publication_schedule`。
  - 返回预览数据、摘要和错误列表。
- `POST /api/schedule/uploads/{upload_id}/commit`
  - 接收管理员对已存上传预览的确认，不接收浏览器端提交的改写刊期行。
  - 通过校验后写入或覆盖该年份 `publication_schedule`。
  - 将上传记录标记为 `committed`。

### 解析服务

新增独立服务负责 PDF 解析和校验：

- 抽取年份，如“二O二六年”或“2026年”。
- 识别月份列、日期、期号和“休刊”。
- 生成每条刊期行：`year`、`publish_date`、`issue_number`、`is_suspended`。
- 提取备注摘要，例如全年出版正报期数、版数、定价、盖章日期。

2026 年样例应解析出：

- 52 条周一计划。
- 49 条出版期。
- 3 条休刊：2026-02-16、2026-02-23、2026-10-05。
- 出版期号 2635-2683，且连续递增。

### 校验规则

预览和保存阶段都执行校验：

- 同一年内 `publish_date` 不重复。
- `publish_date.year` 必须等于刊期表年份。
- 非休刊行必须有正整数期号。
- 休刊行必须没有期号。
- 非休刊期号必须连续递增。
- 行数、出版期数、休刊期数和 PDF 备注摘要不一致时提示管理员核对。
- 如果该年已有 `issues`，禁止覆盖会改变已创建期次的 `issue_number` 或 `publish_date` 的数据。

### 错误处理

- 解析失败返回明确错误，不写入最终刊期表。
- 识别不完整时返回只读预览和错误行；管理员需修正或 OCR 源 PDF 后重新上传，直到校验通过再确认。
- 不对缺失期号、日期或休刊状态做静默猜测。
- 文件类型不是 PDF、空文件、文本不可抽取时直接阻断。
- 保存失败时保留上传记录和错误信息，便于排查。

## 前端设计

新增“刊期表管理”页面：

- 侧边栏新增入口。
- 页面顶部选择年份。
- 展示该年最终刊期表，按月份分组，并在每个月内按出版日期升序列出。
- 提供“上传刊期表 PDF”按钮。

上传后的预览流程：

1. 管理员选择或上传 PDF。
2. 前端调用预览接口并自动显示解析结果。
3. 页面显示解析摘要：年份、总计划周数、出版期数、休刊期数、期号范围、备注。
4. 页面显示只读预览表：日期、期号、是否休刊。
5. 错误行用醒目样式标出。
6. 如解析有误，管理员修正或 OCR 源 PDF 后重新上传。
7. 校验允许时，管理员点击“确认保存”。
8. 保存成功后刷新刊期表页面，并失效 Dashboard、issues、schedule 相关 TanStack Query 缓存。

保存后，现有首页的“一键创建第 X 期”和“选择其他期数补录”继续读取后端刊期数据，不需要重复维护期次来源。

## 权限

- 只有管理员可以上传、确认保存或覆盖年度刊期表。
- 普通操作员可以查看刊期表，但不能改动。

## 测试计划

后端：

- 用 2026 样例文本或 PDF 解析用例验证 52 条计划、49 期、3 休刊、期号范围 2635-2683。
- 校验休刊不占期号、期号连续递增、同年日期唯一。
- 校验已有 `issues` 时禁止破坏性覆盖。
- 校验 commit 后 `GET /api/schedule?year=2026` 返回保存后的数据。

前端：

- 上传 PDF 后显示摘要和预览表。
- 预览表为只读，不能在页面内改写刊期行。
- 解析错误行可见，且校验错误会阻止保存。
- 管理员需修正或 OCR 源 PDF 并重新上传后，才能在校验通过时确认保存。
- 保存成功后刷新刊期表、Dashboard 和可创建期次。

验证命令：

- 后端变更后测试受影响 API 和解析服务。
- 前端变更后运行 `cd frontend && npx tsc --noEmit`。
