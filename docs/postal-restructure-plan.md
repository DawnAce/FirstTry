> 状态：**已落地** —— PR-A(#76) 收款迁财务 · PR-B(#77) 删月度快照 · PR-C(#78) 合并客服工单 · PR-E 工单物理合表。

> 定稿日期：2026-07-21

## 1. 背景与问题

【邮局管理】当前是 1 个一级菜单 + **7 个平级二级菜单**，用户反馈"很冗余、第一眼分不清点哪个"。经排查，冗余是**信息架构层面**的，不是功能缺失：

### 现状 7 个二级菜单

| # | 二级菜单 | 本质 | 主要数据表 |
|---|---------|------|-----------|
| 1 | 投递名册 | 全量投递记录库（动态、CRUD、导入） | `PostalDelivery` |
| 2 | 邮局订报生成 | 上传源文件→解析校验→生成产物→汇入名册 | `SubscriptionBatch/Version/Record/...` |
| 3 | 月度起投明细 | 从名册按「起投月」冻结的快照，导出交邮局 | `PostalDeliveryBatch` / `PostalDeliveryRow` |
| 4 | 投诉工单 | 服务工单（三态流程 + 处理记录时间线） | `PostalComplaint` / `PostalComplaintHandlingRecord` |
| 5 | 改地址 | 服务工单（新旧地址对比 + 应用回写） | `PostalAddressChange` |
| 6 | 回访 | 服务工单（只记录回访结果） | `PostalFollowUp` |
| 7 | 收款发票 | 邮局渠道收款 + 开票 + 挂单 | `PostalFinance` |

### 两处真正的冗余

**冗余一：三个菜单其实是"同一条数据流"被拆成三个入口。**
`订报生成 → 投递名册 → 月度起投明细` 是一条流水线，尤其"月度起投明细"只是名册按月切出来的快照，摆成并列一级入口让用户误以为是三个独立功能。

**冗余二：投诉/改地址/回访 各建一张表、各挂一个菜单。**
参考成熟系统（Zendesk / Salesforce Service Cloud / 有赞客服），客服交互的核心对象只有一个"工单(Ticket/Case)"，靠**类型**字段区分；回访只是工单下的一次跟进活动，从不单独建菜单。当前做法把结构几乎相同的三张表平铺成三个菜单，是冗余的主因。

## 2. 关键认知（讨论中确立）

### 2.1 月度快照是"历史遗留"，且已被订报生成取代
- 月度起投明细（`PostalDeliveryBatch`，PR #39，2026-07 初）诞生时**还没有订报生成**，需要一个"从名册按月冻结导出给邮局"的机制。
- 订报生成（PR #74）作为**核心闭环**后来上线，直接产出给邮局的**汇总表 / 地区分送表 / zip**，并支持保存、投递。
- **业务确认：给邮局的名单永远只来自「邮局订报生成」。** 因此月度快照是重复劳动，整层删除。

### 2.2 「改地址」混了两件不同层次的事
| | 是什么 | 该住在哪 |
|---|---|---|
| ① 地址这个"值"变了 | 订户真实地址更新，并同步在履约的订单 | **客户资料 / 订户档案**（真值源，已建成） |
| ② "有人要求改地址"这件事 | 受理没、处理几次、谁处理、时间线 | **工单** |

二者是一条流水线，不是二选一：
```
改地址工单（受理 + 处理时间线）
        │ 点「应用新地址」
        ▼
客户资料 / 订户档案（地址真值） ──同步──▶ 履约订单
```
**结论：改地址工单不再自存地址真值；「应用」时写回客户资料，复用已有的订单同步逻辑。**

### 2.3 回访不配拥有独立实体/菜单
回访 = 工单时间线里的一条处理记录（渠道 = 电话）。当前投诉工单已有的"处理记录（`PostalComplaintHandlingRecord`）"就是时间线载体。

## 3. 最终结构（已锁定）

**邮局管理：7 个二级菜单 → 3 个；收款发票迁入财务管理。**

```
邮局管理
├─ 投递名册        纯台账 / 查询视图，供客服工单挂靠；不再生成任何名单
├─ 邮局订报生成    保留；唯一产出"给邮局文件"的地方（保存 / 投递都在这）
└─ 客服工单        投诉 + 改地址 + 回访 三类型合并为「工单」，按类型筛选

财务管理
└─ 邮局收款发票    从邮局管理迁入（含后端 API 迁至财务命名空间），与"发票 / 渠道结算"同区
```

### 处置对照表

| 原二级菜单 | 处置 | 说明 |
|-----------|------|------|
| 投递名册 | **保留（降级）** | 变纯台账/查询；表格增"关联工单"列；无"生成名单/快照"动作 |
| 月度起投明细 | **整层删除** | 批次/冻结/快照全砍；名单只来自订报生成 |
| 邮局订报生成 | **保留** | 唯一给邮局出文件的地方，重流程独立入口 |
| 投诉工单 | **合并 → 客服工单** | 类型 = 投诉 |
| 改地址 | **合并 → 客服工单** | 类型 = 改地址；"应用"写回客户资料 |
| 回访 | **合并 → 客服工单** | 类型 = 回访；有对应投诉则并入其时间线，独立回访落成 `类型=回访` 的工单 |
| 收款发票 | **迁出 → 财务管理** | 前后端一并迁移，API 挪至财务命名空间 |

### 三条职责边界
1. **名册管"记录"、订报生成管"给邮局的名单"、工单管"售后过程"**——互不重叠。
2. 改地址：`工单(受理+时间线) → 应用 → 客户资料(真值) → 同步 → 履约订单`。
3. 回访是工单的一种类型：有对应投诉则并入其时间线，独立回访自成一张 `类型=回访` 的工单。

## 4. 客服工单页设计要点

- 顶部按 **类型** 分段筛选：全部 / 投诉 / 改地址 / 回访。
- 列表列：类型、收报人、编号、摘要、状态、处理次数。
- 详情（右侧抽屉/面板）按类型差异化：
  - **投诉**：基本信息 + **处理时间线**（每条含处理动作、结果、是否回访、处理人、时间）+「+ 添加处理/回访记录」。
  - **改地址**：**新旧地址对比** +「应用新地址」按钮 + 时间线；"应用"写回客户资料并同步订单（无匹配兜底见 §6）。
  - **回访**：回访日期 + 结果 + 时间线；可独立存在，也可挂在某投诉工单下。
- 工单↔业务对象关联沿用既有字段（无需新增）：`postal_delivery_id`（按 年度+编号 匹配名册记录）+ `order_id` / `external_order_no`（名册记录挂单则继承），顺此即可追到订单。
- 状态机沿用三态：`待处理 → 处理中 → 已解决`。

## 6. 决策（已定）

1. **收款发票后端归属**：**前后端一并迁移**，`/api/postal/finance` 挪到财务命名空间。动 API 契约，实施时先 grep 全部调用方一起改。
2. **工单是否物理合表**：本轮（PR-C）**先在前端/API 层统一，三表暂存**；**紧接着做 PR-E 物理合表**（不等生产验证，连续做完），设计通用 `PostalTicket` 基表。
3. **回访存量数据落位**：回访作为工单的一种类型。**有对应投诉工单则并入其时间线；独立回访落成 `类型=回访` 的工单**。存量 `PostalFollowUp` 按此迁移。
4. **改地址无匹配兜底**：采**方案 (a)**——客户资料里匹配不到订户时，**仍把新地址写进名册投递记录**，工单标注"已应用（仅名册，未关联客户档案）"并提示后续补建，不拦截工作流。匹配键：订户编号优先，姓名+编号兜底。
5. **删快照前归档**：PR-B 删表前先导出 `PostalDeliveryBatch` / `PostalDeliveryRow` 存量为 json 归档。
